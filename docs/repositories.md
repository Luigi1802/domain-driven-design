# Repositories métier

---

## Introduction

Les repositories sont des **ports** définis par la couche Application pour permettre la persistance et la récupération des agrégats du domaine. Ils représentent des contrats abstraits — des intentions métier — qui seront concrétisés par des adapters techniques (PostgreSQL, MongoDB, in-memory, etc.). Les repositories manipulent des entités et agrégats au niveau métier, jamais des structures de données techniques (pas de tables SQL, pas de documents JSON bruts). Ils garantissent que les invariants métier sont préservés lors de la persistance et de la récupération.

Dans notre système de billetterie événementielle, nous définissons deux repositories métier principaux correspondant à nos deux agrégats : `RepositoryÉvènement` et `RepositoryRéservation`.

---

## 1. RepositoryÉvènement

**Responsabilité métier** :  
Gérer la persistance et la récupération de l'agrégat `Évènement`, qui constitue le cœur du ContexteRéférencement. Ce repository doit garantir les invariants métier de l'agrégat Évènement, notamment l'unicité d'un Évènement pour une combinaison (Artiste, Salle, Date), la cohérence des informations obligatoires, et l'intégrité du stock de Places défini par la capacité de la Salle.

### Opérations métier

| Opération métier | Description (3–4 phrases) | Contraintes / règles métier |
|------------------|---------------------------|-----------------------------|
| **Créer un Évènement** | Persiste un nouvel Évènement dans le référencement après validation de tous les invariants métier. L'opération stocke l'Évènement avec son identifiant unique (EvenementId), ses informations obligatoires (Artiste, Salle, Date, Horaires, OuvertureRéservations, Description), ainsi que l'ensemble de ses CatégoriePlaces et Places. Cette opération est déclenchée uniquement si l'agrégat Évènement a validé tous ses invariants en amont. | **Invariant d'unicité** : Avant de persister, le repository doit vérifier qu'aucun Évènement avec la même combinaison (Artiste, Salle, Date) n'existe déjà. En cas de doublon, l'opération échoue avec une erreur métier. **Cohérence des données** : Tous les champs obligatoires doivent être renseignés. **Intégrité du stock** : Le nombre total de Places doit correspondre à la somme des Places de toutes les CatégoriePlaces et ne doit pas dépasser la capacité maximale de la Salle. |
| **Récupérer un Évènement par EvenementId** | Récupère un Évènement complet à partir de son identifiant métier unique (EvenementId). L'opération retourne l'agrégat reconstitué avec toutes ses entités internes et objets valeur (Artiste, Salle, Horaires, OuvertureRéservations, CatégoriePlaces, Places). Si l'Évènement n'existe pas, l'opération retourne une erreur métier explicite. Cette opération est utilisée par les services applicatifs pour vérifier des invariants ou afficher les informations de l'Évènement. | **Reconstitution complète** : L'agrégat doit être reconstitué dans son intégralité, avec tous ses objets valeur et entités internes, pour garantir que les invariants métier peuvent être appliqués. **Gestion de l'inexistence** : Si l'EvenementId ne correspond à aucun Évènement, le repository retourne une erreur métier (non une exception technique), permettant au service applicatif de gérer ce cas de manière cohérente. |
| **Récupérer tous les Évènements publiés** | Récupère la liste de tous les Évènements dont le statut est `PUBLIÉ` et qui sont visibles dans l'AffichageRéférentiel. Cette opération est utilisée pour afficher la liste des Évènements consultables par les Spectateurs. Le repository peut appliquer des filtres métier (par exemple, filtrer uniquement les Évènements futurs, ou les Évènements d'une Salle donnée). Les Évènements en brouillon ou non publiés ne sont jamais retournés. | **Filtrage métier** : Seuls les Évènements avec un statut `PUBLIÉ` sont retournés. Les Évènements incomplets ou en cours de saisie ne doivent jamais apparaître dans cette liste. **Performance** : Cette opération peut être amenée à retourner un grand nombre d'Évènements. Le repository peut implémenter une pagination, mais cela reste transparent pour le domaine. |
| **Vérifier l'existence d'un Évènement par (Artiste, Salle, Date)** | Vérifie si un Évènement existe déjà pour une combinaison donnée (Artiste, Salle, Date), sans récupérer l'Évènement complet. Cette opération est utilisée pour garantir l'invariant d'unicité lors de la création d'un nouvel Évènement. Elle retourne un booléen (true si un Évènement existe, false sinon). Cette opération est optimisée pour la performance et n'a pas besoin de reconstituer l'agrégat complet. | **Invariant d'unicité** : Cette opération est critique pour garantir l'invariant "Unicité de l'Évènement". Elle doit être exécutée dans une transaction pour éviter les conditions de concurrence (race conditions) où deux Programmateurs tenteraient de créer le même Évènement simultanément. |
| **Mettre à jour les Places disponibles d'un Évènement** | Met à jour le stock de Places disponibles pour un Évènement suite à une Réservation confirmée ou à la libération de Places verrouillées. Cette opération est déclenchée par le ContexteRéservation via un événement métier. Le repository recharge l'agrégat Évènement, applique la modification du stock (décrémentation ou incrémentation du nombre de Places disponibles dans une CatégoriePlace donnée), vérifie que les invariants restent respectés (pas de Places négatives, pas de dépassement de capacité), puis persiste l'état mis à jour. | **Intégrité du stock** : Le nombre de Places disponibles ne peut jamais être négatif. Si une opération tente de réserver plus de Places qu'il n'en reste, elle échoue avec une erreur métier. **Cohérence transactionnelle** : Cette opération doit être effectuée dans une transaction pour garantir la cohérence du stock en cas de réservations concurrentes. Le repository peut utiliser un verrouillage optimiste ou pessimiste selon l'implémentation technique choisie. |

## 2. RepositoryRéservation

**Responsabilité métier** :  
Gérer la persistance et la récupération de l'agrégat `Réservation`, qui est au cœur du processus de réservation du ContexteRéservation. Ce repository doit garantir les invariants métier de l'agrégat Réservation, notamment l'unicité de Place par Évènement, le respect du délai de verrouillage du Panier, et l'obligation de Paiement pour valider une Réservation.

### Opérations métier

| Opération métier | Description (3–4 phrases) | Contraintes / règles métier |
|------------------|---------------------------|-----------------------------|
| **Créer une Réservation** | Persiste une nouvelle Réservation avec son statut initial `EN_ATTENTE_PAIEMENT` suite à l'ajout d'une Place au Panier par un Spectateur. L'opération stocke la Réservation avec son identifiant unique (ReservationId), le Spectateur associé (SpectateurId), l'Évènement cible (EvenementId), la Place sélectionnée, le VerrouillagePlace actif avec son délai d'expiration (15 minutes par défaut), et la Date de création de la Réservation. Cette opération est déclenchée uniquement si l'agrégat Réservation a validé tous ses invariants, notamment que la Place n'est pas déjà verrouillée ou réservée. | **Invariant d'unicité de Place** : Avant de persister, le repository doit vérifier qu'aucune Réservation active (statut `EN_ATTENTE_PAIEMENT` ou `CONFIRMÉE`) n'existe déjà pour cette Place et cet Évènement. En cas de conflit, l'opération échoue avec une erreur métier explicite ("Cette Place est déjà réservée ou verrouillée"). **Intégrité temporelle** : Le délai d'expiration du VerrouillagePlace doit être calculé et stocké correctement. Le repository peut utiliser une colonne timestamp ou un job planifié pour gérer l'expiration automatique des verrouillages. |
| **Récupérer une Réservation par ReservationId** | Récupère une Réservation complète à partir de son identifiant métier unique (ReservationId). L'opération retourne l'agrégat reconstitué avec toutes ses entités internes et objets valeur (Spectateur, Évènement, Place, Panier, VerrouillagePlace, Paiement si présent, Billet si émis). Si la Réservation n'existe pas, l'opération retourne une erreur métier. Cette opération est utilisée par les services applicatifs pour afficher le détail d'une Réservation à un Spectateur ou pour valider un Paiement. | **Reconstitution complète** : L'agrégat doit être reconstitué dans son intégralité, y compris le statut du VerrouillagePlace (actif ou expiré) et les informations de Paiement si elles existent. **Gestion des états** : Le repository doit retourner le statut actuel de la Réservation (`EN_ATTENTE_PAIEMENT`, `CONFIRMÉE`, `EXPIRÉE`, `ANNULÉE`) en fonction de l'état persisté et des règles temporelles (expiration du délai de verrouillage). |
| **Récupérer toutes les Réservations d'un Spectateur** | Récupère la liste de toutes les Réservations associées à un Spectateur donné (identifié par SpectateurId), quel que soit leur statut. Cette opération est utilisée pour afficher l'historique des Réservations d'un Spectateur dans son espace personnel. Le repository peut appliquer des filtres métier (par exemple, uniquement les Réservations confirmées, ou uniquement les Réservations pour des Évènements futurs). Les Réservations sont triées par Date de Réservation décroissante (les plus récentes en premier). | **Filtrage par Spectateur** : Seules les Réservations associées au SpectateurId donné doivent être retournées. **Respect de la vie privée** : Cette opération ne doit retourner que les Réservations du Spectateur authentifié. Le contrôle d'accès est géré en amont par l'adapter REST, mais le repository doit garantir qu'aucune fuite de données n'est possible. |
| **Vérifier si une Place est déjà réservée ou verrouillée** | Vérifie si une Place donnée est déjà réservée (statut `CONFIRMÉE`) ou verrouillée (statut `EN_ATTENTE_PAIEMENT` avec un VerrouillagePlace actif) pour un Évènement donné. Cette opération est critique pour garantir l'invariant "Unicité de Place par Évènement". Elle retourne un booléen (true si la Place est indisponible, false si elle est libre). Cette opération doit être rapide et optimisée, car elle est appelée à chaque tentative de sélection de Place par un Spectateur. | **Invariant d'unicité de Place** : Cette opération est la clé de voûte de l'intégrité du système de réservation. Elle doit être exécutée dans une transaction pour éviter les conditions de concurrence (race conditions) où deux Spectateurs tenteraient de réserver la même Place simultanément. **Gestion de l'expiration** : Un VerrouillagePlace expiré ne doit pas être considéré comme actif. Le repository doit vérifier le délai d'expiration avant de retourner le résultat. |
| **Mettre à jour le statut d'une Réservation** | Met à jour le statut d'une Réservation suite à un événement métier : passage de `EN_ATTENTE_PAIEMENT` à `CONFIRMÉE` après un Paiement réussi, ou passage à `EXPIRÉE` après expiration du délai de verrouillage. Cette opération recharge l'agrégat Réservation, applique la transition d'état en respectant les règles métier (par exemple, une Réservation `CONFIRMÉE` ne peut plus être annulée), puis persiste l'état mis à jour. Si un Billet est émis suite à la confirmation, il est également persisté. | **Transitions d'état valides** : Toutes les transitions d'état ne sont pas permises. Par exemple, une Réservation `EXPIRÉE` ne peut pas revenir à `EN_ATTENTE_PAIEMENT`. Le repository (via l'agrégat) doit garantir que seules les transitions métier valides sont appliquées. **Atomicité** : La mise à jour du statut de la Réservation, l'émission du Billet, et la libération ou confirmation définitive de la Place doivent être effectuées dans une seule transaction atomique pour garantir la cohérence. |
| **Libérer les Places verrouillées expirées** | Identifie toutes les Réservations en statut `EN_ATTENTE_PAIEMENT` dont le VerrouillagePlace a expiré (délai de 15 minutes dépassé), et libère ces Places pour les rendre à nouveau disponibles à la réservation. Cette opération est exécutée périodiquement par un job planifié (batch) ou déclenchée par un événement temporel. Pour chaque Réservation expirée, le repository met à jour son statut à `EXPIRÉE`, libère la Place associée, et peut éventuellement notifier le Spectateur via le ContexteNotification. | **Identification des expirations** : Le repository doit parcourir toutes les Réservations actives et comparer leur délai d'expiration avec l'heure courante. Cette opération peut être coûteuse en performance ; elle doit être optimisée (index sur les colonnes de date, traitement par lot). **Cohérence transactionnelle** : La libération des Places et la mise à jour des statuts doivent être effectuées dans une transaction pour éviter les incohérences. **Impact sur le stock** : La libération d'une Place doit également mettre à jour le stock de Places disponibles dans l'Évènement correspondant. |

## Notes d'implémentation (conceptuelles)

Ces repositories sont des **contrats abstraits** définis par la couche Application. Ils ne contiennent aucun détail technique (pas de SQL, pas de code). Les adapters techniques (par exemple, `RepositoryÉvènementPostgreSQL` ou `RepositoryRéservationMongoDB`) fourniront les implémentations concrètes en respectant ces contrats.

**Choix de persistance envisagés** :
- **PostgreSQL** : Base de données relationnelle adaptée pour garantir l'intégrité transactionnelle et les contraintes d'unicité via des clés primaires et index uniques. Les invariants comme l'unicité de l'Évènement ou de la Place peuvent être renforcés au niveau de la base de données.
- **MongoDB** : Base de données orientée documents, adaptée si l'on souhaite persister les agrégats sous forme de documents JSON complets. Nécessite une gestion applicative plus rigoureuse des contraintes d'unicité et de concurrence.
- **In-memory** : Adapter temporaire utilisé uniquement pour les tests unitaires ou l'exploration du domaine, sans persistance réelle.

**Gestion de la concurrence** :
Les opérations critiques (création de Réservation, vérification de disponibilité de Place) doivent être protégées contre les conditions de concurrence. Les adapters peuvent utiliser :
- **Verrouillage optimiste** : version de l'agrégat incrémentée à chaque modification, détection des conflits lors de la sauvegarde.
- **Verrouillage pessimiste** : verrouillage de la ligne en base de données (SELECT FOR UPDATE) pendant la transaction.

Le choix entre ces stratégies dépend du volume de trafic attendu et des performances requises. Ce choix est transparent pour le domaine et l'application, car il est encapsulé dans les adapters.
